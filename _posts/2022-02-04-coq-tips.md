# Coq이란?
프로그램이 올바르게 작성되었다는 것을 검증하는 방식으로 여러가지가 있는데, 대표적인 세가지로 [model checking](https://en.wikipedia.org/wiki/Model_checking), [interactive theorem proving](https://en.wikipedia.org/wiki/Proof_assistant), 그리고 [static analysis](https://en.wikipedia.org/wiki/Static_program_analysis)가 있다. 이 방법론들은 각각이 장단점을 갖고 있어서, 어느 하나가 다른 것보다 절대적으로 우월하다고는 할 수 없다. coq이라는 것은 그 중에서 interactive theorem proving(이하 ITP) 동네에서 전형적으로 사용되고 있는 툴, 즉 증명보조기(proof assistant)이다. ITP의 기본 아이디어는 명제<->타입, 증명<->프로그램으로 대표되는 대응관계인 소위 [curry-howard isomorphism](https://en.wikipedia.org/wiki/Curry%E2%80%93Howard_correspondence)에 근거한다. 이것이 뭐냐면, 우리가 수학에서 어떤 "명제"를 "증명"한다는 것은, 그야말로 일개 프로그래머가 주어진 "타입"에 들어맞는 적당한 "프로그램"을 작성하는 것과 동등한 행위라는 것이다. 

그렇다면 이걸 어떻게 프로그램 검증에 이용할 수 있느냐? 우리가 원하는 성질(가령, "프로그램 add에 인자로 a, b를 넣으면 그 출력값 c는 a+b이다" 따위)을 "타입"으로 인코딩한 후, 그 타입에 해당하는 coq "프로그램"(여기서의 프로그램은 앞선 예시의 add 프로그램과 완전 다른 것) 을 아무거나 작성하면 된다. curry-howard isomorphism에 의하면, 이는 "명제"를 "증명"한 셈이기 때문이다. 이런게 가능하기 위해서는 당연히 "타입"이라는 것이 c언어의 그것처럼 단순해서는 안되고, "for all ~", "there exists ~" 따위를 표현할 수 있을 정도로 강력해야 한다. 그러한 타입시스템의 대표적인 예로, 스웨덴의 논리학자인 마틴 뢰프가 지대한 공헌을 한 [dependent types](https://en.wikipedia.org/wiki/Dependent_type)가 있다. 내가 이해하기로는, 프랑스의 전산학자인 Thierry Coquand이 이를 dependent types -> Calculus of Constructions(CoC) -> Calculus of Inductive Constructions(CiC) -> Gallina -> Coq 순으로 확장한 것이 바로 Coq이다 [(참고)](https://www.cs.cornell.edu/courses/cs6120/2022sp/blog/coc/). 

실제로 Coq을 이용하여 그 유명한 [4색 정리](https://en.wikipedia.org/wiki/Four_color_theorem)를 증명하기도 했고, [CompCert](https://compcert.org/compcert-C.html)라고 최적화가 올바르다고 수학적으로 검증된 컴파일러가 만들어지기도 했을만큼, Coq은 단순한 연구거리에 불과한 것이 아니라 실제로 쓸모가 있다는 것이 점차 밝혀지고 있다. 가장 낙천적인 관점에서, Coq은 미래에 수학자들의 펜과 종이를 대체할 수 있을 것이다. 그런 의미에서 수학자들은 이미 이론적으로 "일종의" 프로그래머이지만, Coq(혹은 그런 류의 어느 기타 증명보조기라도)이 수학자들의 원자재가 되는 날에는 수학자들은 "말 그대로" 프로그래머일 것이다. 실제로 러시아의 필즈상 수상자 Voevodsky에 의해 수학의 토대를 프로그래밍의 토대로 갈아엎는 [프로젝트](https://en.wikipedia.org/wiki/Univalent_foundations)가 진행 중이다.

## 설치
linux 기준으로 설명한다. coq을 커맨드라인 환경에서 이용할 경우 ocaml이라는 언어의 패키지 매니저인 opam으로 설치하면 편한데, coq 자체가 ocaml로 개발됐기 때문이다.
자세한 설치 방법은 [공식 홈페이지](https://coq.inria.fr/opam-using.html)을 참고하면 된다.
시스템에 opam만 있으면 다음처럼 손쉽게 coq을 설치할 수 있다. (참고로, opam에서 coq 소스를 갖고와서 빌드하기 때문에 시간이 좀 걸린다.)

<h5 a><strong><code>shell</code></strong></h5>

``` shell
$ opam pin add coq 8.12.0
$ coqc -v
> The Coq Proof Assistant, version 8.12.0 (September 2022)
```

위처럼 coqc -v를 시스템에 입력했을 때 뭐가 그럴 듯하게 뜨면 coq이 제대로 설치된 것이다.

## 개발환경

vim에서 coq을 사용할 경우, [coqtail](https://github.com/whonore/Coqtail) 플러그인을 이용하면 된다. 자세한 설치방법은 coqtail 레포지토리를 참고하면 된다. 나는 [vim-plug](https://github.com/junegunn/vim-plug)라는 vim 플러그인 매니저를 이용하여 설치하였다.

우선 vim-plug가 있다고 가정하고, .vimrc 파일의 적당한 곳에 다음을 추가시킨다.
<h5 a><strong><code>~/.vimrc</code></strong></h5>

``` vim
call plug#begin('~/.vim/plugged')
Plug 'whonore/Coqtail'
call plug#end()
```
그런 후에, shell에 다음을 입력하면 설치가 완료된다.
<h5 a><strong><code>shell</code></strong></h5>

``` shell
$ vim +PlugInstall +qa
```
만약 이 과정에서 문제가 있다면, vim이 python3을 지원하지 않아서일 가능성이 있다. 이 경우 vim을 python3을 지원하도록 다시 인스톨해보자.

coqtail 플러그인이 잘 설치됐는지 확인하기 위해, 임시 coq파일(.v 확장자)를 vim으로 실행시킨 후,
``` shell
$ vim hello.v
```
\cc를 눌러보자. "\\", "c", "c" 키를 순서대로 세번에 걸쳐 연달아 치면된다. 그랬을때 vim 창이 세개로 분리되면서 우하단의 창에 와인잔 비슷한 그림이 나오면 coqtail이 성공적으로 설치된 것이다.

# 시작하기 전에..
## 명제와 증명
Coq을 사용하기 전에 최소한 알아야 할 기본적 이론에 대해서 간단히 짚고 넘어가고자 한다. 수학자들은 명제를 증명하는 사람들이다. 수학에는 정수론, 대수학, 위상수학 등 다양한 세부분야가 있다. 이들 각 분야에서는 각자만의 고유한 수학적 대상에 대해 탐구한다. 예를 들어, 정수론자들은 산술적 대상들에 대해 탐구하고, 위상수학자들은 도넛에 대해 탐구한다. 어떻게 탐구하느냐? 페르마의 마지막 정리 따위의 "명제"를 만들고 그걸 "증명"하면 된다. 그럼 도대체 "명제"와 "증명"이란건 무엇이냐? 그걸 탐구의 대상으로 삼는게 바로 수리논리학자들이다. 그러니까 위상수학자들에게 도넛과 머그컵 같은게 바로 논리학자들에게 명제고 증명이 되겠다, 이 말이다.

우선, **명제**란 무엇이냐? 중학교 수학시간에 "명제와 집합" 단원에서 명제논리라는 것을 배운다. 대충 "지금 비가온다", "대한민국의 수도는 서울이다" 따위가 명제고, 명제들을 "and", "or", "implies" 등등으로 엮어서 복잡한 명제를 만들 수 있다가 요지이다. 사람의 언어에서 이러한 요소들의 뜻은 자명하다. 예를 들어 "지금 비가오거나, *또는* 대한민국의 수도는 서울이다"라는 명제의 뜻은 직관적으로 이해할 수 있다. 그럼 그런 우리의 직관을 수학적으로 정의할 수 있을까? 한가지 자연스런 방법은 *진리표*로 정의하는 것이다. 가령 "지금 비가온다" 등의 의미는 0 아니면 1 중에 하나고, "and"의 의미는 (1,1)을 넣으면 1을 출력하고 (0,0), (0,1), (1,0)을 넣으면 0을 출력하는 함수로 정의하는 것이다. 그런데 이런 방식의 정의는 우리에게 가장 익숙한 정의일 뿐, 정의하기 나름이다. Coq의 관점에서 명제의 의미는 그것이 *어떻게 증명되는지*로 정의된다.

그럼 **증명**이란 무엇이냐? 이것도 정의하기 나름이다. 실제로 말이되는 증명 시스템이 여러개 있다. 한가지 미묘한점은, 수학자들과 전산학자들이 바라보는 증명의 정의가 약간 다르다는 것이다. (문학적으로 표현하면, 수학자들은 신의 관점에서 증명을 바라보고 전산학자들은 인간의 관점에서 바라본다.) 어찌됐건 증명이 뭔지는 "표준"만이 있을 뿐 정답이란건 없다. Coq의 관점에서 증명이란 것은 소위 증명나무(proof tree)를 만드는 일이다. 이런 방식의 증명시스템은 독일의 논리학자 겐첸이 처음 도입한 [자연연역(Natural Deduction)](https://en.wikipedia.org/wiki/Natural_deduction)으로 거슬러 올라간다.

## 자연연역
Coq의 기반이되는 증명시스템인 CoC(Calculus of Constructions)은 자연연역 스타일을 따른다. CoC에 관해서는 나중에 알아보기로 하고, 자연연역이 맛만 보도록 하자. 자연연역이라는 것은 쉽게 말해서 복잡한 명제를 증명하기 위해 미리 구성해놓은 작은 증명조각들이다.

예를 들어 명제 `A /\ B` (즉, A and B)를 증명한다고 생각해보자. "자연"연역이니 가장 "자연"스런 증명은 `A`를 증명하고, `B`를 증명하면 된다. 이를 자연연역에서는 다음처럼 `/\-I`라는 이름의 "증명조각"으로 정의해놓는다.
```
 A true  B true
----------------- /\-I
   A /\ B true
```
위의 표기가 도대체 무슨 의미인가? "아래에 있는걸 증명하려면 위에 있는 두개를 증명해야한다"로 읽으면 된다.
위 증명조각이 어떻게 복잡한 명제를 증명하는 데에 "이용"되는지 알면 명확해진다. 

```
 A true  B true
----------------- /\-I
   A /\ B true          C true
  ------------------------------- /\-I
     (A /\ B) /\ C true

```
위 그림의 핵심은, 두 증명조각을 합쳤다는 것이다. 이런 방식으로 작은 증명조각을 "합쳐서" 큰 증명을 만들어나가는 것이 자연연역의 핵심이다. 생긴 걸 보면 나무를 거꾸로 뒤집은 모양이다. 그래서 "증명나무"라 부른다.
그렇다면 위 증명나무는 완성된 증명인가? 아니다. ...를 증명하기 위해 ...가 필요하다는 것을 보여줄 뿐, ...가 실제로 '참'이라는 것을 증명되는 것은 아니다. ...가 참이기 위해서는, A, B, C가 참이라는 전제가 필요하다. 증명없이 무언가를 참이라고 간주하는 것을 자연연역에서는 다음과 같이 표기한다. 즉, 아래를 증명하기 위해 위에 아무 것도 필요없다.
```
-------    ------    --------
A true     B true     C true
```
위의 세 규칙만 있으면 위의 증명을 완성할 수 있다.

```
------- -------
 A true  B true
----------------- /\-I  ------
   A /\ B true          C true
  ------------------------------- /\-I
     (A /\ B) /\ C true

```
이렇게 모든 가지가 닫혀있는 나무를 우리는 "증명"이라고 한다. 자연연역이라는 것은 이런 규칙들을 모아놓은 규칙바구니이다.
지금까지는 수많은 규칙들 중 하나만 살펴본 것이고, 자연연역에서는 아래처럼 다양한 요소들마다 규칙을 미리 정의해 놓는다.

* `T` : truth
* `F` : falsehood
* `A`, `B`, ... : variable
* `/\` : and
* `\/` : or
* `->` : implies

정리하자면, 위의 요소들을 조합해서 만든 문장이 "명제"이고, 규칙들을 조합하여 닫힌증명나무를 만드는 것이 바로 "증명"이다.

# coq 시작하기
coq을 실행시키면 다음과 같이 세개의 창이 뜬다. 
1. Code - .v 확장자의 Coq 프로그램을 수정할 수 있는 텍스트 에디터
2. Goal - 현재 증명하고 있는 proof tree의 상태
3. Info - 일종의 stdout

용어
1. 증명상태 proof state
2. 명령어 tactic
3. 목표 Goal

Coq으로 증명하는 것은 기본적으로 proof theory에서 [natural deduction](https://en.wikipedia.org/wiki/Natural_deduction) 방식으로 proof tree를 이용해 증명을 하는 것과 비슷하다.

위에서 했던 예제를 coq으로 하면 다음과 같다.

``` coq
Variables A B C : Prop.

Proposition foo : A -> B -> C -> (A /\ B) /\ C.
Proof.
intros H1 H2 H3.
split.
- split.
  * apply H1.
  * apply H2.
- apply H3.
Qed.
```
한줄한줄씩 살펴보자.
``` coq
Variables A B C : Prop.

Proposition foo : A -> B -> C -> (A /\ B) /\ C.
```
우선 `A`, `B`, `C`라는 변수를 선언하고, `A -> B -> C -> (A /\ B) /\ C`라는 명제를 증명하려고 한다. 주목할 점은, 마치 `A -> B -> C -> (A /\ B) /\ C` 타입의 변수 `foo`를 선언하는 것 같이 생겼다는 것이다. 사실, 그렇게 생긴게 아니라 진짜 그렇다!

위 코드를 실행시키면 다음의 증명나무가 Goal창에 생성될 것이다. 즉, 우리의 목표는 아래 증명나무의 윗부분을 Coq 명령어를 통해 채워놓는 것이다.
```
-----------------------------
A -> B -> C -> (A /\ B) /\ C
```
다음의 명령어를 실행시키면 위의 증명상태를 바꿀 수 있다:
``` coq
intros H1 H2 H3.
```
`intros` 명령어는 `A -> B` 꼴의 목표를 가진 증명상태에서 `A`를 가정함으로써 `B` 꼴의 목표를 가지도록 변환시켜준다. 현재 목표는 `A -> B -> C -> (A /\ B) /\ C`이니, 3개의 가정을 intros하면 `A`, `B`, `C`가 전제로 빠지고 목표는 `(A /\ B) /\ C`가 되는 것이다. 이렇게 함으로써 이제 증명상태는 다음과 같다:
```
      ----  ---  ---
       A     B    C
         ( ... )
-----------------------------
     (A /\ B) /\ C
```
보다시피 `A`, `B`, `C`가 공리처럼 맨 위에서 증명을 마무리하기만을 기다리고 있다. 이제 우리는 이 간극을 메꿔주기만 하면 된다. 이제 무엇을 해야할까? 남은 목표를 보자. `(A /\ B) /\ C`를 증명해야한다. 이는 가장 바깥에서 봤을 때 `(A /\ B)`와 `C`가 `/\`로 엮인 명제이다. 앞의 자연연역에서 이걸 어떻게 했는지 생각해보자. 왼쪽과 오른쪽 각각 가지를 뻗었었다. 이를 `split` 명령어로 수행할 수 있다:

``` coq
split.
```
이제 증명상태는 다음과 같이 바뀐다:
<h5 a><strong><code>증명상태</code></strong></h5>

```
  ----  --- ---   ---   ---  ---
   A    B    C     A     B    C
    ( . . . )       ( . . .)
    ---------       --------
    (A /\ B)           C
--------------------------------------
          (A /\ B) /\ C
```
<h5 a><strong><code>부분목표1</code></strong></h5>

```
  ----  --- ---
   A    B    C
    ( . . . )
    ---------
    (A /\ B)
```
<h5 a><strong><code>부분목표2</code></strong></h5>

```
---   ---  ---
 A     B    C
   ( . . .)
   --------
      C
```

여기서 구멍이 두개 생긴 것을 볼 수 있다. 즉 원래 목표에서 두개의 부분목표가 생긴 것이다. 이를 `부분목표1`(왼쪽구멍), `부분목표2`(오른쪽 구멍)라고 하고 각각 증명을 '`-`' 표시로 구분해준 뒤 완성해주면 된다. 먼저 `부분목표1` - 즉, `(A /\ B)` - 의 증명부터 살펴보자:
``` coq
- split.
  * apply H1.
  * apply H2.
```
앞에서와 마찬가지로, `A /\ B` 또한 `A`와 `B`가 `/\`로 엮여있는 형태이니 이전과 정확히 똑같은 원리로 `split`을 해주는 것이다. 그러면 `부분목표1`로부터 두개의 부분목표들이 갈라져나와 `부분목표1-1`(왼쪽), `부분목표1-2`(오른쪽)를 형성한다:
<h5 a><strong><code>부분목표1</code></strong></h5>

```
   - - -   -  -  -
   A B C   A  B  C
   (...)     (...)
    ---       ---
     A         B
    -------------
       (A /\ B)
```
`apply`명령어은 전제에 갖고 있는 증명을 이용하여 현재 목표의 증명을 완성시켜준다. `부분목표1`을 생각해보면, 전제에 `A`, `B`, `C`를 각각 `H1`, `H2`, `H3`이라는 이름으로 갖고 있고, `A`를 증명하는 것이 목표이다. 이때 `apply H1`을 해주면 다음과 같이 부분목표의 증명상태의 구멍이 메꿔진다:
<h5 a><strong><code>부분목표1-1</code></strong></h5>

```
  ----  --- ---
   A    B    C     apply H1   -----
    ( . . . )       =====>      A
    ---------
       A
```
`부분목표1-1`과 마찬가지로 `부분목표1-2`에 대해서도 똑같이 `apply H2`를 수행함으로써 `B`를 증명할 수 있다:
<h5 a><strong><code>부분목표1-2</code></strong></h5>

```
  ----  --- ---
   A    B    C     apply H2   -----
    ( . . . )       =====>      B
    ---------
       B
```
이렇게 `부분목표1-1`과 `부분목표1-2`의 구멍을 채워넣음으로써 `부분목표1`의 구멍도 자동으로 메꿔지게 된다:
<h5 a><strong><code>부분목표1</code></strong></h5>

```
   - - -   -  -  -
   A B C   A  B  C             ---    ---
   (...)     (...)              A      B
    ---       ---     ====>    ----------
     A         B                (A /\ B)
    -------------
       (A /\ B)
```
이제 최초의 증명상태를 생각해보자. 남은 구멍은 오로지 `부분목표2` 뿐이다. `부분목표2` 또한 `apply H3` 명령으로 손쉽게 해결할 수 있다:
``` coq
- apply H3.
```
<h5 a><strong><code>부분목표2</code></strong></h5>

```
  ----  --- ---
   A    B    C     apply H3   -----
    ( . . . )       =====>      C
    ---------
       C
```
<h5 a><strong><code>증명상태</code></strong></h5>

```
  ----  --- ---   ---   ---  ---
   A    B    C     A     B    C                    ---  ---       ---
    ( . . . )       ( . . .)                        A    B         C
    ---------       --------            ====>     ----------     -----
    (A /\ B)           C                           (A /\ B)        C
-------------------------------------        ----------------------------
          (A /\ B) /\ C                              (A /\ B) /\ C
```
